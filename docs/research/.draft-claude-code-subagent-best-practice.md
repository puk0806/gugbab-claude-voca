# Claude Code 서브에이전트 활용 Best Practice -- 리서치 보고서 (초안)

**작성일**: 2026-03-26 | **주제**: Claude Code 서브에이전트 활용 best practice | **검색 소스**: WebSearch, WebFetch

## 요약 (Executive Summary)

Claude Code의 서브에이전트는 독립된 컨텍스트 윈도우에서 특화된 작업을 수행하는 전문 AI 어시스턴트이다. 2025-2026년 기준으로 서브에이전트 활용은 단순 코드 자동완성을 넘어 병렬 리서치, 보안 감사, 테스트 자동화, 멀티 에이전트 협업 등 프로덕션 수준의 워크플로우로 진화했다. 학술 연구에서는 멀티 에이전트 오케스트레이션이 단일 에이전트 대비 100% 실행 가능한 권고사항 생성률(단일 에이전트 1.7%)을 달성했으며, Anthropic 내부에서는 16개 에이전트가 병렬로 10만 줄 규모의 C 컴파일러를 Rust로 구축하는 데 성공했다. 핵심 best practice는 (1) 3-4개 이하의 집중된 서브에이전트 운용, (2) 읽기 전용 도구 제한을 통한 안전성 확보, (3) 모델 티어링(Opus-Sonnet-Haiku)을 통한 비용 최적화, (4) 퍼시스턴트 메모리를 활용한 학습 축적이다.

## 1. 논문 & 학술 연구

### 1.1 Multi-Agent LLM Orchestration for Incident Response (2026.01)
- **출처**: https://arxiv.org/abs/2511.15755
- **신뢰도**: High (peer-reviewed, 348 controlled trials)
- **핵심 내용**: MyAntFarm.ai 프레임워크를 통해 단일 에이전트 vs 멀티 에이전트 시스템을 348회 통제 실험으로 비교. 멀티 에이전트가 100% 실행 가능한 권고사항 생성률을 달성(단일 에이전트 1.7%). 행동 특이성 80배, 솔루션 정확도 140배 향상. 양 아키텍처 모두 이해 속도는 ~40초로 유사하여, 핵심 가치는 속도가 아닌 신뢰성에 있음을 입증.
- **시사점**: 멀티 에이전트 오케스트레이션은 성능 향상이 아닌 '프로덕션 준비성'의 전제 조건으로 재정의되어야 한다. 결정적(deterministic) 행동이 SLA 커밋을 가능케 함.

### 1.2 The Orchestration of Multi-Agent Systems: Architectures, Protocols, and Enterprise Adoption (2026.01)
- **출처**: https://arxiv.org/html/2601.13671v1
- **신뢰도**: High (comprehensive survey)
- **핵심 내용**: 오케스트레이션을 자율 에이전트를 "목표 지향적 집단"으로 변환하는 제어 평면(control plane)으로 정의. 5개 핵심 유닛(계획/정책, 실행/제어, 상태/지식, 품질/운영, 통신)으로 구성. Worker/Service/Support 에이전트의 3가지 역할 분류 체계 제시. MCP와 A2A 프로토콜을 표준 통신 수단으로 분석.
- **시사점**: 보험 인수심사 95% 정확도, 모기지 처리 20배 가속/80% 비용 절감, 소프트웨어 현대화 50% 개발 시간 단축 등 실질적 엔터프라이즈 성과 보고.

### 1.3 Multi-Agent Collaboration via Evolving Orchestration (2025.10)
- **출처**: https://arxiv.org/abs/2505.19591
- **신뢰도**: High (OpenReview peer-reviewed)
- **핵심 내용**: 중앙 집중형 오케스트레이터가 강화학습을 통해 에이전트 시퀀싱과 우선순위를 적응적으로 조정하는 "퍼펫티어(puppeteer)" 패러다임 제안. 정적 파이프라인 대비 동적 태스크 상태 기반 에이전트 조율의 우위를 입증.
- **시사점**: 서브에이전트 호출 순서와 병렬/순차 결정을 자동화하는 것이 차세대 오케스트레이션의 핵심 방향.

### 1.4 Multi-Agent Collaboration Mechanisms: A Survey of LLMs (2025.01)
- **출처**: https://arxiv.org/html/2501.06322v1
- **신뢰도**: High (comprehensive survey)
- **핵심 내용**: 협업 메커니즘을 5가지 차원으로 분류하는 확장 가능한 프레임워크 제시. 5G/6G 네트워크, 인더스트리 5.0, QA, 소셜 설정 등 다양한 도메인에서의 적용 사례를 탐색.
- **시사점**: 멀티 에이전트 협업의 범용적 분류 체계는 Claude Code 서브에이전트 설계 시 역할 정의에 참고할 수 있음.

## 3. 기업 도입 사례 & 공식 가이드

### 3.1 Anthropic 공식 문서: 서브에이전트 가이드
- **출처**: https://code.claude.com/docs/en/sub-agents
- **신뢰도**: High (공식 1차 자료)
- **핵심 내용**: 서브에이전트는 `.claude/agents/` 또는 `~/.claude/agents/`에 YAML frontmatter를 가진 마크다운 파일로 정의. 내장 서브에이전트(Explore/Plan/General-purpose)와 커스텀 서브에이전트를 지원. 도구 제한(tools/disallowedTools), 모델 선택(sonnet/opus/haiku/inherit), 퍼미션 모드(default/acceptEdits/dontAsk/bypassPermissions/plan), 퍼시스턴트 메모리(user/project/local), 후크(PreToolUse/PostToolUse/Stop), MCP 서버 스코핑, 백그라운드 실행 등 포괄적 설정 지원.
- **교훈**: 서브에이전트는 3-4개로 제한하고, 각각이 하나의 특정 태스크에 탁월하도록 설계하며, 상세한 description으로 자동 위임을 유도해야 함.

### 3.2 Anthropic 공식 문서: 에이전트 팀
- **출처**: https://code.claude.com/docs/en/agent-teams
- **신뢰도**: High (공식 1차 자료)
- **핵심 내용**: 서브에이전트와 달리 에이전트 팀은 독립된 Claude Code 인스턴스들이 공유 태스크 리스트와 메시지 시스템을 통해 직접 소통. 팀 리드가 조율하고, 팀원들이 자율적으로 태스크를 클레임. 3-5명 팀원이 최적이며, 팀원당 5-6개 태스크가 적절. Split pane(tmux/iTerm2) 또는 In-process 디스플레이 모드 지원.
- **교훈**: 서브에이전트는 결과만 필요할 때, 에이전트 팀은 상호 토론/도전/협업이 필요할 때 사용. 파일 충돌 방지를 위해 팀원별 파일 소유권 분리 필수.

- **도입 배경**: 16개 Claude Opus 4.6 에이전트가 독립 Docker 컨테이너에서 병렬로 Rust 기반 C 컴파일러를 구축하는 대규모 실험. 약 2,000회 Claude Code 세션, $20,000 API 비용 투입.

## 4. 종합 인사이트 -- Claude Code 서브에이전트 Best Practice

### 4.1 설계 원칙

1. **집중된 전문화 (Focused Specialization)**: 서브에이전트는 하나의 명확한 태스크에 특화시킨다. 3-4개를 초과하면 어떤 에이전트를 호출할지 결정하는 오버헤드가 생산성을 저하시킨다.

2. **최소 권한 원칙 (Least Privilege)**: 읽기 전용 태스크에는 Write/Edit 도구를 제거하고, DB 접근에는 PreToolUse 후크로 SELECT만 허용하는 등 필요한 도구만 부여한다.

3. **명확한 description 작성**: Claude가 자동 위임 여부를 description 기반으로 결정하므로, "use proactively" 등의 문구를 포함해 적극적 위임을 유도한다.

4. **모델 티어링 (Model Tiering)**: 메인 세션은 Opus로, 탐색/검색은 Haiku로, 분석은 Sonnet으로 운용하여 비용 대비 품질을 최적화한다.

### 4.2 아키텍처 패턴

1. **병렬 리서치 패턴**: 독립적인 조사 영역을 별도 서브에이전트에 할당하고, 메인 에이전트가 결과를 종합한다.
   ```
   메인 -> [서브에이전트A: 보안 분석] + [서브에이전트B: 성능 분석] + [서브에이전트C: 테스트 커버리지]
   ```

2. **체이닝 패턴**: 서브에이전트를 순차적으로 연결하여 파이프라인을 구성한다. 코드 리뷰 -> 최적화 -> 테스트 등.

3. **에이전트 팀 패턴**: 상호 소통이 필요한 복잡한 작업에는 서브에이전트 대신 에이전트 팀을 사용한다. 공유 태스크 리스트, 직접 메시징, 경쟁 가설 검증 등.

4. **격리 패턴 (Worktree Isolation)**: `isolation: worktree` 설정으로 서브에이전트에 독립된 git worktree를 제공하여 파일 충돌을 원천 방지한다.

### 4.3 운영 가이드라인

1. **컨텍스트 관리**: 대량 출력(테스트 실행, 로그 분석)은 서브에이전트에 격리하여 메인 컨텍스트를 보존한다.

2. **퍼시스턴트 메모리 활용**: `memory: project` 스코프로 서브에이전트가 세션 간 학습을 축적하게 하여 점진적으로 효과를 높인다. MEMORY.md에 코드베이스 패턴, 반복 이슈, 아키텍처 결정을 기록하도록 시스템 프롬프트에 명시한다.

3. **후크 기반 품질 게이트**: PreToolUse/PostToolUse 후크로 SQL 쓰기 차단, 린터 자동 실행 등 조건부 검증을 적용한다.

4. **비용 제어**: Haiku 기반 Explore 서브에이전트로 탐색 비용을 절감하고, 에이전트 팀 사용 시 토큰이 팀원 수에 선형 비례함을 인지한다.

5. **팀 규모**: 에이전트 팀은 3-5명이 최적이며, 팀원당 5-6개 태스크가 적절하다. 그 이상은 조율 오버헤드가 이득을 상쇄한다.
